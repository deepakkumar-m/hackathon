📋 Scenario

🌊 Traffic spike incoming!

A load test script sends 200 concurrent requests to your application. With only 1–2 replicas, the app crashes under load.

Pods: 2/2 Crashing
Response time: > 30s | Error rate: 85%
Status: Service Unavailable (503)

📝 Steps to Complete

1

Scale Replicas

Immediately scale up to handle the traffic:

kubectl scale deployment myapp --replicas=5 -n your-namespace
2

Configure Horizontal Pod Autoscaler (HPA)

Set up automatic scaling based on CPU usage:

kubectl autoscale deployment myapp \
  --cpu-percent=50 \
  --min=2 \
  --max=10 \
  -n your-namespace
3

Monitor with Grafana

Open Grafana ↗ and watch:

  • 📈 Request rate and response times
  • 📊 CPU and memory usage per pod
  • 🔄 Pod count changes from HPA
  • ❌ Error rates dropping
4

Verify Stability

Use Rancher ↗ to verify all pods are running and the app is responding correctly. Error rate should drop to near 0%.

🎯 What You Learn

📐

Horizontal Scaling

Add more pods to handle load, not bigger pods (horizontal vs vertical).

📊

Observability

You can't fix what you can't see. Monitoring is critical for production.

🤖

HPA (Auto-Scaling)

Let Kubernetes automatically scale based on metrics — no manual intervention.

🔥

Production Troubleshooting

Real-world skills: diagnose, scale, monitor, stabilize under pressure.
