📋 Scenario

⚠️ The pipeline passes, but the app fails!

The Java application has a hardcoded database password. The pipeline builds fine, but when deployed, the pod crashes because the database secret isn't provided via Kubernetes.

Pod Status: CrashLoopBackOff
Error: Connection refused — DB_PASSWORD not set as environment variable

📝 Steps to Complete

1

Create a Kubernetes Secret

Go to Rancher ↗ or use kubectl to create a secret in your namespace:

kubectl create secret generic db-credentials \
  --from-literal=DB_PASSWORD=mySecurePassword123 \
  -n your-namespace
2

Modify the Helm Chart

Update values.yaml to reference the secret:

env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: DB_PASSWORD
3

Update Deployment Template

Ensure the deployment template in templates/deployment.yaml injects the environment variable into the container.

4

Deploy & Verify

Push changes, sync via ArgoCD ↗, and verify the pod starts successfully in Rancher ↗.

🎯 What You Learn

🔐

Never Hardcode Secrets

Passwords, API keys, and tokens must never be stored in source code.

🗝️

Kubernetes Secrets

Use K8s Secrets to store sensitive data and inject via environment variables.

⎈

Helm Values Mastery

Modify Helm values to control application configuration declaratively.

💉

Environment Injection

Understand how secrets flow from K8s → Pod → Container → Application.
