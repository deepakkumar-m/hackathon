📋 Scenario

You are given a Java Spring Boot application, a Dockerfile, and a Helm chart. Your task is to follow the traditional (manual) process that most teams use before CI/CD automation.

This challenge exposes the pain of manual deployments so you appreciate what automation brings.

📝 Steps to Complete

1

Clone the Repository

Go to Bitbucket ↗ and clone the Java application repository to your local machine.

git clone https://bitbucket.org/gbme/hackathon-app.git
2

Make a Code Change

Open the Java application and modify the welcome message or version string. This simulates a developer making changes.

3

Build Docker Image via Jenkins

Go to Jenkins ↗ and trigger a build. Jenkins will:

  • Pull your code from Bitbucket
  • Build the Docker image using the Dockerfile
  • Tag the image with a version (e.g., 1.0)
4

Push Image to JFrog Registry

After a successful build, manually push the image to JFrog Artifactory ↗.

docker push your-jfrog.jfrog.io/docker-local/myapp:1.0
5

Update Helm Values

Open the Helm chart's values.yaml and manually update the image tag to match the new version you just pushed.

image:
  repository: your-jfrog.jfrog.io/docker-local/myapp
  tag: "1.0"  # ← Update this manually
6

Deploy via ArgoCD

Go to ArgoCD ↗ and sync the application. ArgoCD will deploy your updated Helm chart to Kubernetes.

🎯 What You Learn

🔨

What is CI?

Continuous Integration — automatically building and testing code on every change.

🚀

What is CD?

Continuous Delivery/Deployment — automatically deploying tested code to environments.

😫

Why Manual is Painful

Every manual step is a chance for error, delay, and inconsistency.
