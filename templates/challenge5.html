📋 Scenario

💥 Pod keeps crashing — OOMKilled!

The application intentionally consumes excessive memory. The namespace has a resource quota, and the pod doesn't have proper resource limits configured. Result: OOMKilled.

Pod Status: OOMKilled
Last State: Terminated - Reason: OOMKilled
Exit Code: 137

📝 Steps to Complete

1

Check Pod Logs

Go to Rancher ↗ or use kubectl:

kubectl logs <pod-name> -n your-namespace
kubectl logs <pod-name> -n your-namespace --previous
2

Describe the Pod

Check the pod events to see the OOMKilled reason:

kubectl describe pod <pod-name> -n your-namespace

# Look for:
# Last State:  Terminated
#   Reason:    OOMKilled
#   Exit Code: 137
3

Add Resource Requests & Limits

Update values.yaml in the Helm chart:

resources:
  requests:
    cpu: 250m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
4

Deploy & Monitor

Push the changes, sync via ArgoCD, and monitor in Rancher ↗. The pod should now be stable.

💡 Key Concepts

Concept
Meaning
requests.memory
Minimum memory the pod needs to be scheduled
limits.memory
Maximum memory — if exceeded, K8s kills the pod (OOMKilled)
requests.cpu
Minimum CPU guaranteed for the pod
limits.cpu
Maximum CPU — if exceeded, pod gets throttled (not killed)
ResourceQuota
Namespace-level limit on total resources consumed

🎯 What You Learn

📏

Requests vs Limits

Requests guarantee resources; limits cap them. Both are essential.

🏗️

Resource Quotas

Namespaces can enforce resource budgets to prevent runaway pods.

⚖️

Right-Sizing

Too little → crashes. Too much → waste. Proper sizing is critical.
