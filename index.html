üê≥ Understanding the Dockerfile

You have a simple Java application in Bitbucket. Your first mission is to understand how applications are containerized using Docker. Read the Dockerfile line by line and understand what each instruction does.

üéØ Objectives

  • Clone the Java application repository from Bitbucket
  • Read and understand each Dockerfile instruction
  • Identify the base image, build stages, and exposed ports
  • Complete the interactive quiz below
FROM - Base image your container starts from (e.g., openjdk:17-slim)
COPY - Copies files from your local machine into the container
RUN - Executes commands during image build (e.g., mvn clean package)
EXPOSE - Documents which port the container listens on
ENTRYPOINT / CMD - The command that runs when the container starts
üìÑ Dockerfile
1 FROM maven:3.8.6-openjdk-17-slim AS builder
2 WORKDIR /app
3 COPY pom.xml .
4 RUN mvn dependency:go-offline
5 COPY src ./src
6 RUN mvn clean package -DskipTests
7 FROM openjdk:17-slim
8 WORKDIR /app
9 COPY --from=builder /app/target/*.jar app.jar
10 EXPOSE 8080
11 ENTRYPOINT ["java", "-jar", "app.jar"]

üîÑ GitOps with ArgoCD

ArgoCD is a GitOps continuous delivery tool for Kubernetes. It watches your Git repo and automatically syncs your Helm charts to the cluster. No more manual kubectl apply!

üéØ Objectives

  • Understand GitOps principles and ArgoCD architecture
  • Create an ArgoCD Application manifest
  • Connect your Helm chart repo to ArgoCD
  • Trigger a sync and observe automatic deployment
ArgoCD follows the GitOps pattern: Git is the single source of truth
Application CRD tells ArgoCD what to deploy and where
ArgoCD auto-detects drift between Git and cluster state
Sync policies: Manual vs Auto-Sync
